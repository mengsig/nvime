OVERVIEW                                                        *nvime-overview*

nvime routes Claude Code and Codex CLI through explicit review/docs and edit
lanes. The edit lane is intentionally narrow: one visual range or one
Tree-sitter function, one current file, written intent, reviewed diff.
The selected range may be blank when the user's intent is to generate or insert
new current-file code/text.

Requires Neovim 0.10 or newer.

nvime is a Neovim guardrail, not a host security sandbox.

COMMANDS                                                        *nvime-commands*

                                                        *:Nvime*
:Nvime
        Open the nvime command center. This is a Mason-style floating console
        with saved chat and selection discussion counts, running-agent state,
        block-highlighted tabs, action rows, recent sessions, a dim backdrop,
        provider/native-resume badges, and in-buffer ? / g? help. Dashboard
        number keys switch tabs; number shortcuts still open visible rows in
        scoped pickers.

                                                        *:NvimeChat*
:NvimeChat
        Open the picker for general chat/review conversations. Press n to start
        a fresh conversation, or open an older conversation from the list. Each
        conversation opens as one floating nvime://chat/<id> scratch buffer with
        the prompt at the bottom, focused in normal mode. Status lives in the
        float title/footer instead of being written into the transcript. The
        buffer contains only messages, responses, and the active [provider]$
        prompt. nvime guards the buffer so only the live prompt line can be
        changed while typing.
        Press i or o from anywhere in the chat to return to the prompt; press
        <CR> on the prompt to submit a chat/review message.
        nvime keeps a transcript and native provider session id per general
        conversation. Same-provider follow-ups resume the native Claude/Codex
        conversation. Switching providers sends that conversation's nvime
        transcript so the newly selected provider has shared context.
        Review/docs runs reuse a stable temporary workspace for the provider
        session so native resumes resolve in the same project path.
        While an agent is running, nvime flushes start/tool/command activity
        and provider-supplied reasoning summaries into the float footer/status
        area instead of appending them to the transcript. Final answers and
        provider errors still appear in the buffer.
        By default, this lane may run shell commands, including curl, may use
        web fetch/search tools, and may create or update Markdown files only.
        Providers run in a temporary workspace copy, and nvime syncs only *.md
        and *.markdown files back into the real repo.

                                                        *:NvimeChats*
:NvimeChats [chat|ask|edit]
        Open the picker for general chat conversations or highlighted-code
        Ask/Edit discussions. The picker has a new-session row plus existing
        sessions, so <leader>nc, <leader>nq, and <leader>ne act as an
        intermediate step before either starting fresh or continuing an older
        conversation. General chats are persisted by default in
        .nvime/chat-sessions.json. Selection discussions are persisted by
        default in .nvime/selection-sessions.json. In the picker, press 1-9 to
        open a numbered row, <CR> to open the cursor row, dd to delete the
        current row, or visual-select rows with V and press d to delete several.
        If a hidden session finishes, ui.completion = "notify" reports it with
        vim.notify instead of reopening the float. Set ui.completion = "open"
        for pop-open behavior. :NvimeLast or <leader>nn reopens the last used
        conversation.

                                                        *:NvimeReview*
:NvimeReview [claude|codex] [prompt]
        Run a review/docs lane. The agent may inspect context, run shell
        commands when configured, and write Markdown documentation files. nvime
        does not apply source/config edits from this lane.

                                                        *:NvimeLast*
:NvimeLast
        Reopen the last used general chat or Ask/Edit discussion. If no session
        has been used in the current Neovim process, it opens the most recently
        updated persisted chat or selection discussion.

                                                        *:NvimeAsk*
:'<,'>NvimeAsk [claude|codex] {question}
:NvimeAsk [claude|codex] {question}
        Ask the read-only side agent about a selected range or Tree-sitter
        function. This is for questions such as "does this look right?". It does
        not open a diff. Ask and Edit keep workflow discussions per selected
        file/range, separate from general chat. If the same range already has a
        discussion, visual <leader>nq can continue it or start a new native
        provider session. The visual chooser also offers same-file and recent
        repo discussions, so a newly highlighted range can reuse context from a
        persisted conversation. If no question is supplied inline, nvime asks
        on the guarded prompt line instead of opening a popup prompt.
        After an Ask answer, the prompt remains armed for that selection.
        Follow-up questions use Ask again; repair requests such as "please
        proceed", "fix this", "suggest the diff", or "apply that change" launch
        Edit on the same range without reselecting it. If an Ask response
        already contains NVIME_DIFF, NVIME_REPLACEMENT, or a current-file
        unified diff, nvime opens it as an inline review instead of leaving it
        as plain chat text. When an inline review opens, the selection
        conversation float closes so the file and accept/reject mappings are
        immediately visible; reopen the discussion with <leader>ne or
        <leader>nq.

                                                        *:NvimeEdit*
:'<,'>NvimeEdit [claude|codex] {intent}
:NvimeEdit [claude|codex] {intent}
        Request one focused edit. With a range, the range is used. Without a
        range, nvime asks Tree-sitter for the function at the cursor. If the
        agent reports that no change is needed, or returns an identical patch,
        no diff is opened. Question-shaped and review-shaped edit requests are
        routed to |:NvimeAsk|. The latest Ask result for the same file/range is
        included as context for the next edit request.
        Blank selected ranges are valid insertion targets. Non-code current
        file ranges, such as .gitignore entries or config snippets, are also
        valid as long as the patch stays inside the selected range.
        Edit prompts prefer minimal NVIME_DIFF hunks for existing text,
        especially Markdown and large selections, instead of reproducing the
        whole selected file.
        Starting Edit after Ask forces the input prompt to the edit lane, so the
        next submission can produce a reviewed patch instead of remaining in
        read-only Ask mode.
        Visual <leader>ne can continue an exact range match, a same-file
        discussion, or a recent repo discussion with the provider's native
        resume id, when available, or start a fresh nvime/provider session.
        If no intent is supplied inline, nvime asks on the guarded prompt line
        instead of opening a popup prompt.

	                                                        *:NvimeAccept*
:NvimeAccept
	        Accept the current pending inline diff block in the target file.

	                                                        *:NvimeAccept!*
:NvimeAccept!
	        Force-accept the current pending or conflicted inline diff block.
	        This bypasses the live-content guard and writes a
	        block_force_applied audit event.

                                                        *:NvimeCancel*
:NvimeCancel
        Cancel the active nvime agent run.

                                                        *:NvimeDisable*
:NvimeDisable
        Cancel running nvime agents, restore wrapped Neovim APIs, and stop
        persisting sessions while disabled. Stored wrapper references captured
        by other plugins before disable are not rewritten.

                                                        *:NvimeEnable*
:NvimeEnable
        Re-enable nvime after |:NvimeDisable| and reinstall guard wrappers when
        guard.enabled is true.

                                                        *:NvimeReject*
:NvimeReject
        Reject the current pending inline diff block in the target file.

                                                        *:NvimeDiff*
:NvimeDiff
        Open the active diff in a two-pane review workspace: full proposed
        result on the left, the live editable source buffer with inline diff
        overlays on the right.

                                                        *:NvimeAudit*
:NvimeAudit
:NvimeAudit raw
:NvimeAudit summary [days]
:NvimeAudit forces
        Inspect the audit log. The bare command (and `raw`) opens
        .nvime/audit.jsonl directly. `summary` opens a digest float that
        groups events by lane/provider, surfaces risky events
        (block_force_applied, block_conflict, plan_step_rollback,
        blocked), and ranks the most-touched files. The default window is
        7 days; pass an explicit number for a different range. `forces`
        opens a focused review of every force-accepted diff block — the
        single class of action that bypasses the live-content guard, and
        the place where unnoticed changes can hide.

                                                        *:NvimeAttribute*
:NvimeAttribute [show|hide|toggle]
        Show the attribution entry for the line under the cursor: which
        plan and step authored it, the patch worker's RATIONALE, the
        critic verdict, and the timestamp. Without an arg, prints the
        entry to vim.notify. `show` paints inline virtual-text annotations
        on every line covered by an attribution anchor in the current
        buffer; `hide` clears them; `toggle` flips. Entries are persisted
        to .nvime/attribution.json (in the git root); the ledger is
        capped at attribution.max entries (default 500) with oldest
        rotated out.

                                                        *:NvimeRecap*
:NvimeRecap [claude|codex] [--cached] [<rev>..<rev>]
        Reverse-direction "explain my changes" lane. Computes a git diff
        (working tree by default; `--cached` for staged changes; an
        A..B range for any commit interval), spawns the plan-lane agent
        in a temp workspace, and asks it to write a plan.md narrative
        explaining what changed, why, and what is untested. Output lands
        under .nvime/plans/recap-<short-hash>/ and is auto-opened in the
        plan view. Catches the case where work happened outside nvime
        and you want to re-establish understanding without retroactive
        vibe-coding.

                                                        *:NvimeProvider*
:NvimeProvider [claude|codex]
        Show or set the default provider used by nvime commands and chat.

                                                        *:NvimePlan*
:NvimePlan
:NvimePlan new [intent]
:NvimePlan open <id>
:NvimePlan run <id> [step]
:NvimePlan refresh
:NvimePlan delete <id>
:NvimePlan discuss <id>
:NvimePlan add-test <id> <step>
:NvimePlan reset-session <id> [provider]
        Drive the plan author and executor lanes (see |nvime-plan|). With no
        args, opens the plan picker. `new` drafts a plan via the plan author
        agent — restricted to writing files under .nvime/plans/<id>/. `open`
        opens the plan view buffer. `run` executes a step (the next pending
        step if no step argument is given) by opening the step's file/range
        and routing through |:NvimeEdit| with the existing inline diff
        review.

Inline diff mappings in the target file:
        Pending diffs are shown as readable change blocks: proposed green lines
        above, current red lines below, and one compact controls header per
        block. Large contiguous replacements are split into smaller visual
        blocks by default. Review units remain line-level internally.
        :NvimeDiff
                open the proposed/editable review workspace
        ]n      next unresolved line
        [n      previous unresolved line
        ]b      next visual change block
        [b      previous visual change block
        ga      accept the current visual change block
        v_ga    accept unresolved changed lines touched by the visual range
	        gA      accept all unresolved blocks
	        gA!     force-accept all unresolved/conflicted blocks
        gu      undo the most recently accepted block when unchanged
	        gb      reject the current visual change block
        v_gb    reject unresolved changed lines touched by the visual range
        gB      reject all unresolved blocks
        gr      reject the current unresolved line (alias)
        v_gr    reject unresolved changed lines touched by the visual range
                (alias of v_gb)
        gR      reject all unresolved blocks (alias of gB)
        gX      reject the current visual change block (alias of gb)
        gc      discuss accepted/rejected/current/unresolved lines with the
                edit agent

Diff review workspace mappings:
        e       focus the editable source pane
        ]b/[b   next/previous visual change block
        ]n/[n   next/previous unresolved line
        ga/gb   accept/reject the current visual block
        gu      undo the most recently accepted block when unchanged
        gA/gB   accept/reject all unresolved blocks
        gc      discuss active review state with the edit agent
        r       refresh the review panes
        q       close the review workspace from either pane and return to
                the tab you came from

KEYMAPS                                                          *nvime-keymaps*

Default keymaps use <leader>n:
        <leader>n<Space>
                       open dashboard command center
        <leader>nc     open general chat conversation picker
        <leader>nr     run review/docs
        <leader>nq     ask picker: new current-function session or old chat
        visual <leader>nq
                       ask about selected range
        <leader>ne     edit picker: new current-function session or old chat
        visual <leader>ne
                       edit selected range
        <leader>na     open audit log
        <leader>nd     discuss active inline diff state
        <leader>nv     open active diff review workspace
        <leader>nn     reopen last used nvime conversation
        <leader>np     choose Claude or Codex
        <leader>nP     open the plan picker

Command center mappings:
        <CR>, o, i      open the cursor row
        1-5            switch dashboard tabs
        picker 1-9     open a numbered visible session row
        n              start a fresh chat/review session
        a              start Ask from the current function on dashboard
        e              start an Edit discussion from the current function
        dd             delete the selected session row
        V ... d        delete selected session rows
        r              refresh the command center
        ?, g?          open or close the in-buffer help
        q, <Esc>       close the command center

Chat buffer mappings:
        i, I           focus the prompt (cursor near start of input)
        a, A, o, O     focus the prompt (cursor at end of line)
        <Esc>          focus the scrollback (leave the prompt)
        ?              choose a configured prompt template
        prompt <CR>    submit the current message
        <C-c>          cancel the active chat/review agent
        p              cycle provider for the next chat prompt
        <Tab>          cycle provider for the next chat prompt
        P              choose Claude or Codex for the next chat prompt
        q              close the floating chat window

Selection workflow mappings:
        i, I           focus the prompt (cursor near start of input)
        a, A, o, O     focus the prompt (cursor at end of line)
        <Esc>          focus the scrollback (leave the prompt)
        ?              choose a configured prompt template
        prompt <CR>    submit the current question/intent
        <C-c>          cancel the active Ask/Edit/Discuss agent
        p              cycle provider for this discussion's next prompt
        <Tab>          cycle provider for this discussion's next prompt
        P              choose Claude or Codex for this discussion's next prompt
        q              close the floating selection window

Native provider resume:
        General chat and focused Ask/Edit discussions persist provider
        sessions. nvime captures Claude stream-json session_id values and Codex
        JSON session_id values. Follow-ups in the same nvime session use
        claude --resume or codex exec resume. Starting a new nvime session
        creates a separate provider conversation.

Disable defaults with:
>
	        require("nvime").setup({
	          keys = { enabled = false },
	        })
<

Force re-initialization:
>
        require("nvime").setup({
          force = true,
        })
<
        Repeated identical setup() calls are ignored. Use force = true only
        when you intentionally want to rebuild wrappers, commands, and keymaps.

CONFIGURATION                                              *nvime-configuration*

lazy.nvim local install:
>
        {
          "mengsig/nvime",
          name = "nvime",
          opts = {},
        }
<

Common options:
>
        require("nvime").setup({
          provider = "claude",
          ui = {
            layout = "float",
            side = "right",
            completion = "notify",
            ascii_icons = false,
            icons = {},
            spinner_frames = nil,
          },
          keys = {
            enabled = true,
            prefix = "<leader>n",
          },
          audit = {
            enabled = true,
            path = nil,
            log_prompts = false,
          },
          review = {
            allow_shell = true,
            allow_web = true,
            allow_markdown_writes = true,
          },
          selection = {
            allow_shell = true,
            allow_web = true,
          },
          chat = {
            max_history_messages = 24,
          },
          sessions = {
            enabled = true,
            path = nil,
            chat_path = nil,
            max = 100,
          },
          diff = {
            max_visual_block_lines = 12,
          },
          plan = {
            enabled = true,
            dir = nil,                  -- defaults to <git-root>/.nvime/plans
            auto_open = true,           -- open the plan view after authoring
            auto_in_progress = true,    -- mark a step in_progress on execute
            inject_context_chars = 480, -- plan-context block budget per step
            devils_advocate = true,     -- read-only critic on every step diff
            test_file = nil,            -- defaults to auto-detect, e.g. tests/headless_spec.lua
            test_runner = nil,          -- defaults to auto-detect from project markers
            session_continuity = "plan",-- "plan" (steps share session) | "none"
          },
          diff = {
            max_visual_block_lines = 12,
            devils_advocate = false,    -- enable critic for ALL edit-lane diffs (off by default)
          },
        })
<

Unknown config keys and obvious type mismatches are reported with vim.notify
when setup runs. |:checkhealth| nvime checks the active provider commands,
writable state directory, guard wrapper state, Tree-sitter parsers for loaded
buffers, and blocked guard events from the last day.

Default audit path:
	        .nvime/audit.jsonl in the current git root, or
	        stdpath("state")/nvime/audit.jsonl when there is no git root.

PROTOCOL                                                      *nvime-protocol*

The edit lane accepts exactly one machine-readable response block:

>
        NVIME_NO_CHANGE
        <brief explanation>

        NVIME_REPLACEMENT
        ```
        <full replacement for the selected range only>
        ```

        NVIME_DIFF
        ```diff
        --- a/path
        +++ b/path
        @@ -line,count +line,count @@
        <minimal changed hunk lines only>
        ```
<

Top-level edit rules from the prompt contract:
        - You are a constrained patch worker, not a reviewer.
        - Return exactly one machine-readable response block.
        - You may only propose changes for the selected range in the current
          file.
        - Do not edit files directly.
        - Prefer NVIME_DIFF for any change to existing nonblank text.
        - NVIME_DIFF must include --- a/path, +++ b/path, and ranged @@
          headers.

Top-level perf rules from the prompt contract:
        - Focus on computational cost and scalability.
        - If you cannot prove a real win with numbers, return
          NVIME_NO_CHANGE.
        - Use Bash scratch files under /tmp and never write inside the user's
          repository.
        - Only produce NVIME_DIFF when the candidate is correct and faster.
        - Perf mode may emit one BENCH line before the NVIME_* marker.

NVIME_DIFF should include current-file headers and ranged hunk headers. If a
diff body is missing ranged headers, nvime tries to anchor the old lines inside
the selected range and rebuild a current-file hunk. If that anchor fails, no
patch opens. Question-shaped edit intents such as "review", "verify",
"nitpick", or "does this look right?" are routed to |:NvimeAsk| unless they also
include an explicit patch verb such as "fix", "add", "remove", "replace",
"implement", or "update".

The full protocol is documented in PROTOCOL.md at the repository root.

PLAN                                                                *nvime-plan*

The plan lane decomposes a code change into ordered steps that are executed
through the existing edit lane. The agent that drafts the plan runs in a
hardened sub-lane:

    plan_author   read tools + Bash + Web + Write/Edit/MultiEdit, but the
                  workspace is a temp copy whose only files synced back to the
                  repo are those under .nvime/plans/<plan-id>/. Anything written
                  outside that directory is silently dropped on agent exit.

    plan_exec    not its own agent lane; instead, opening a step's file/range
                  and routing through |:NvimeEdit|. The step's intent is
                  prefixed with a constant-size plan-context block (plan id,
                  why, step position, acceptance) so the patch worker has the
                  architectural context without paying multi-step token cost.

Plan storage:
    .nvime/plans/<NNNN-slug>/plan.json    machine-readable contract
    .nvime/plans/<NNNN-slug>/plan.md      human-readable narrative
    .nvime/plans/index.json               index used by the picker

Plan view buffer mappings (nvime://plan/<id>):
    <CR>        execute the step under the cursor (opens file, fires edit lane)
    gE          edit-then-execute: open a compose buffer with the would-be
                intent + plan-context prefilled; tweak, then <C-s> to fire
    gx          mark the step under the cursor as done
    gp          mark the step as pending
    gB          mark the step as blocked
    gT          run the step's tests in a split terminal
    gW          scaffold a regression test for the step (test author lane)
    gN          reset the plan's provider session (next step starts fresh)
    ]s, [s      next/previous step
    gA          execute the next pending step
    gd          discuss/refine the step (re-enters plan author lane)
    gr          replan from this point
    o           open the step's file
    c           copy the step intent
    q, <Esc>    close the plan view
    ?           show keymap help

Plan executor enhancements:
    - Range anchors. Each step may carry a `range_anchor` (1-3 verbatim
      lines from the original target). At execute-time nvime searches the
      file for the anchor and re-anchors the line range to wherever the
      content has drifted to. This survives the case where earlier steps
      modify the same file and shift the line numbers of later steps.
    - Edit-then-execute. Press gE on a step to open a compose buffer with
      the would-be intent + plan-context prefilled. Edit freely (correct a
      stale line range, narrow the request, add a hint), then <C-s> to
      actually run the patch worker against your edited intent.
    - Auto test-runner detection. The test scaffolder (gW) and the default
      runner picked when a step has no `tests` field are auto-detected from
      project markers: Cargo.toml → cargo test, build.zig → zig build test,
      go.mod → go test ./..., pyproject.toml/pytest.ini → pytest -q,
      package.json → npm test, pom.xml → mvn test, build.gradle →
      gradle test, CMakeLists.txt → ctest, Makefile → make test, and
      `scripts/test` always wins when present. Override via
      plan.test_runner.
    - Before running step.tests nvime saves the modified buffer to disk so
      tools that read from disk (stylua --check, ./scripts/test, rg, etc.)
      see the new content rather than the pre-patch file.
    - When all blocks of a step's diff are resolved, nvime auto-runs
      step.tests and offers Done / Pending / Cancel on green; on red it
      offers Rollback / Blocked / Pending / Cancel. Rollback restores the
      file to its pre-step content using the snapshot the diff session
      captured.
    - With plan.devils_advocate = true (default), every plan-step diff is
      shadowed by a read-only critic agent that returns APPROVE / FLAG /
      REJECT in one sentence. The verdict appears as a colored banner at
      the top of the diff review and is advisory only — it never blocks
      the user.
    - Per-plan session continuity (plan.session_continuity = "plan",
      default): all steps of a plan share a provider conversation, so
      step 5 retains the architectural context built up by steps 1-4.
      The session id is captured on plan_author exit, stored in
      plan.json under provider_sessions, and rotates as the agent emits
      new ids on resume. Press gN in the plan view (or run
      :NvimePlan reset-session <id> [provider]) to clear the captured id
      and force the next step to start fresh — useful when the
      conversation has accumulated too much context or gone off-track.
      Set plan.session_continuity = "none" to revert to fresh-each-time.

Plan picker buffer mappings (nvime://plans):
    <CR>            open the plan under the cursor
    n, N, <C-n>     draft a new plan (compose buffer)
    R               refine the plan under the cursor
    dd              delete the plan under the cursor
    r               refresh
    q, <Esc>        close

Plan compose buffer mappings (nvime://plan/compose):
    <C-s>           submit the draft and start the plan author agent
    <C-c>           cancel a running agent (or close if idle)
    q, <Esc>        close the float (your draft is preserved)
    gC              reset the buffer to the template
    ?               show keymap help
    `gx` is intentionally a no-op inside compose so the global vim.ui.open
    handler can never accidentally launch an external opener.

The full plan author response protocol is documented in PROTOCOL.md.

POLICY                                                            *nvime-policy*

Direct claude/codex launches through common Neovim process APIs are blocked.
nvime-owned launches are allowed and audited.

Known limitations:
        - external terminals are out of scope
        - a hostile plugin can bypass Lua wrappers
        - terminal commands typed inside an already-running shell are hard to
          inspect before execution

 vim:tw=78:ts=8:ft=help:norl:
